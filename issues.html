Pile of photos — implementation notes & known issues

This is an unlinked page documenting how the pile-of-photos effect works and the technical trade-offs behind it. It is not part of the site navigation.

The goal

Lay several photos on top of each other so the eyes line up across all of them, but each photo sits at a different angle. The result is a stack of tilted prints on a background, with the background showing through the fanned gaps and around the edges — like a pile of physical prints dealt onto a table.

Is it technically possible? Yes.

It is fully achievable in the browser with the Canvas 2D API — no server, no third-party code, consistent with the rest of the site. The key realisation, exactly as suggested in the feature request, is that you do not need to physically re-cut each source photo. You can fake the rotated cutout by clipping to a rectangle that is axis-aligned in the output frame and then rotating the whole thing. Relative to the original photo's pixel grid, that clip rectangle is rotated — which is precisely the desired "cut a rectangle not aligned with the original photo" effect.

How each print is drawn

  1. Align by the eyes. From the two marked eye points we compute the usual similarity transform (rotate + uniform scale + translate) that puts the eye midpoint on a fixed spot with a fixed inter-eye distance.
  2. Pivot on the eyes. We translate the canvas origin to that fixed eye spot and rotate by the photo's own deal angle. Because the pivot is the eye midpoint, the eyes stay locked in place no matter how far the print is tilted.
  3. Clip to a print rectangle. A rectangle a little smaller than the frame (a fraction of its size) is used as a clip region, positioned so the face sits naturally inside it. Only the pixels inside this rotated rectangle are drawn, so the photo becomes a bounded "print" rather than filling the whole frame.
  4. Draw the aligned photo inside the clip. The source image is drawn with the alignment transform, but only what falls inside the print rectangle survives.
  5. Border & shadow. A thin white border and a soft drop shadow are painted behind each print so overlapping prints stay distinguishable and read with a little depth.

Every frame paints the background once and then stacks all the prints, bringing a different photo to the top each frame, so the whole pile is visible throughout the animation. The relevant code lives in drawPrint and renderPileFrame in assets/js/align.js.

Known limitations & trade-offs

  • Prints overlap heavily. Because the eyes are locked to the same point, the prints coincide almost entirely — only the fanned corners and edges of the ones underneath peek out. This is inherent to "eyes line up" and is what produces the pile look; raising the Pile tilt spreads the fan wider so more of each print shows.
  • The print may include the photo's own background. The cutout is a fixed rectangle around the face, so a tightly-cropped source photo can leave part of the print showing its original surroundings rather than a clean edge. There is no per-photo background removal (that would need a segmentation model, which the site deliberately avoids to keep everything local and dependency-free).
  • Print size is a global fraction of the frame, not tuned per photo. Faces framed very differently across photos are not individually re-cropped. A future enhancement could size each print from its inter-eye distance.
  • Shadows are per-print, not true occlusion. Each print casts its own soft shadow; the renderer does not compute real contact shadows between specific overlapping prints.
  • Eyes must be marked on a photo for it to appear in the pile, the same as for the normal animation.
